# Summary — Agentic AI

Topics covered in `wiki/Agentic AI/` and their key points.

![[Pasted image 20260630112321.png|293]]


| Topic | Key Points |
| ----- | ---------- |
| Definition | - An agent is an LLM embedded in a plan-act-observe-adapt loop that iteratively pursues goals using tools<br>- Four-level spectrum: chatbots (passive) → workflows (fixed steps) → single agent (model decides execution path) → multi-agent systems (coordinated specialists with persistent memory)<br>- Level 3 key distinction: the model itself decides which tools to call and in what order, rather than following a fixed script<br>- Level 4 adds: persistent memory across sessions, skills (task-specific instruction folders), MCPs for tool connectivity, human-in-the-loop checkpoints<br>- The plan-act-observe-adapt loop gives agents their power but also introduces failure modes absent in single-shot LLM usage |
| Failures | - Three structural failure modes: infinite loops, hallucinated planning, unsafe tool use<br>- Infinite loops: no termination conditions and no progress tracking → agent retries indefinitely; mitigate with max step limits<br>- Hallucinated planning: agent assumes capabilities or tools it doesn't have; mitigate with explicit tool schemas and plan validation before execution<br>- Unsafe tool use: over-privileged tools execute destructive actions; mitigate with least-agency principle and approval workflows<br>- Failures are predictable and structural — caused by insufficient engineering discipline, not model quality |
| Agent OS | - An Agent OS is a coordination layer between agents and infrastructure, analogous to an OS kernel<br>- "Genius goldfish" problem: agents are highly capable but stateless and uncoordinated without supervision<br>- Six kernel components: Scheduler (task priority), Memory Manager (short/long-term), Tool Manager (sandboxed access registry), Identity Manager (auth + least privilege), Observability (logs every decision), Guardrails (input/output validation)<br>- Tool Manager provides a canonical registry of available tools — directly prevents hallucinated planning<br>- Identity Manager uses short-lived tokens and audit trails to enforce least-agency at infrastructure level<br>- An Agent OS transforms agents from clever prototypes into trustworthy infrastructure |
| Memory | - Four memory types: working memory (context window), semantic memory (knowledge bases / vector DBs), procedural memory (skills), episodic memory (learnings from past runs)<br>- Working memory = context window; limited in size (up to ~1M tokens for the largest models)<br>- Semantic memory is always present in context; contains general knowledge loaded into the session<br>- Procedural memory uses progressive disclosure — only skills matching the current task are loaded, not all skills at once<br>- Episodic memory uses distillation: saves only important information from previous runs, not everything<br>- Not every agent needs all four types; a simple reflex agent may only need working memory |
| Collaboration (MCP + ADK) | - Two distinct problems to solve: connectivity (how agents access external tools) and orchestration (how agents are structured and coordinated)<br>- MCP (Model Context Protocol, by Anthropic): open standard for LLM-tool connectivity via JSON-RPC; three primitives: Tools, Resources, Prompts<br>- MCP is model-agnostic and reusable — write a server wrapper once, any MCP-compatible client can use it regardless of which LLM powers it<br>- ADK (Agent Development Kit, by Google): Python framework for building agents; three agent types: LLM-driven, Workflow (hardcoded control flow), Custom<br>- ADK's Runner/yield mechanism separates reasoning (Agent) from control (Runner), making behavior predictable and debuggable<br>- MCP and ADK compose: an ADK agent can register MCP servers as tool sources — structured orchestration plus standardised connectivity |
| Building & Deploying | - Four deployment paths, each with a different trade-off between speed, governance, and control<br>- Managed platform (hyperscaler): full governance, monitoring, audit trail, enterprise-ready; slower to prototype<br>- Foundation model infra: optimized for development speed; vendor lock-in, no governance, no enterprise integration<br>- Custom infra: full infrastructure flexibility and control; requires more engineering work<br>- No-code (drag-and-drop): accessible to non-developers, agent is one node in an automation; no governance or control<br>- No path gives all three: speed, governance, and control always involve trade-offs |
| Scaling | - Scaling agentic AI is a system design problem, not a model capability problem<br>- Single-agent ownership of all state causes: noisy context, cascading failures, non-linear cost growth, degraded reasoning quality<br>- Error propagation: a wrong assumption early in a task infects the entire run — autonomy amplifies risk<br>- Solution: multi-agent decomposition with bounded responsibility — failures stay contained within components<br>- Horizontal scaling (add new agents): modular capabilities but more coordination overhead<br>- Vertical scaling (add tools/sub-agents within one agent): less coordination but more per-agent latency and complexity<br>- Rule: split reusable/independent capabilities horizontally; embed tightly coupled capabilities vertically |
| Harness Engineering | - The harness is the full scaffolding around an AI agent: context, tool permissions, execution, and observation layers<br>- Core mindset: every mistake the model makes becomes a constraint in the harness — iteratively tighten until behavior is reliable<br>- Four harness layers: Context (what model sees), Tool/Permission (what it can do), Execution (how actions run), Observation (how failures are caught)<br>- Knowing which layer to fix is the core skill: a repeated mistake gets a rule (context), a safety issue gets a hook (execution), over-access gets a permission restriction<br>- Spectrum from vibe coding (prompt-level) to agentic engineering (harness-level) — the bottleneck is requirements engineering |
| Context Engineering | - Context engineering is the information design layer of the harness — structuring what the model sees to maximize output quality<br>- Six context window content types: Instructions, Examples, Memory, State, Retrieved Knowledge, Tool Outputs<br>- Key design decision: static context (always loaded, e.g. CLAUDE.md) vs. dynamic context (loaded on demand, e.g. skills)<br>- Skills are the primary dynamic context mechanism — lightweight generalist agent that loads specialist knowledge only when needed<br>- Four primitives shared across tools: standing context, on-demand tasks, specialized agents, automated triggers<br>- Naming trap: GitHub Copilot calls on-demand tasks "prompts" and capability packages "skills" — Claude Code calls on-demand tasks "skills"; these are different things with the same word |
| Observability | - Three telemetry pillars: Logs (what happened), Traces (how it happened), Metrics (how much / how well — latency, token counts, error rates)<br>- Three observability primitives: Run (single LLM call), Trace (full agent execution), Thread (multi-turn conversation across sessions)<br>- Traces replace stack traces as the source of truth — non-deterministic agents can't be debugged from code paths alone<br>- OpenTelemetry (OTEL) is the vendor-neutral standard: Spans = Runs, distributed Trace IDs propagate across microservices and MCP servers via OTEL headers<br>- Two signal categories: explicit (error rate, latency, cost — objective) and implicit (user frustration, refusals, forgetting — detected by trained classifiers)<br>- Security: apply data protection policies at the log-group level to redact PII — distinct from prompt-level guardrails |
| Evaluation | - Production is the discovery phase — agent behavior is non-deterministic, so correctness is found in the wild, not proven before release<br>- Three evaluation scopes: single-step (unit test for one decision), full-turn/trace (end-to-end trajectory), multi-turn/thread (realistic conversation flows with memory)<br>- Three evaluation timings: offline (pre-ship regression testing), online (real-time LLM-as-judge on live traces), ad hoc (exploratory analysis triggered by user feedback)<br>- Evals and monitoring are complementary: evals catch known failure modes; monitoring catches long-tail unknowns that only emerge at scale<br>- Production traces automatically become the eval dataset — a virtuous cycle from observability infrastructure to evaluation |
